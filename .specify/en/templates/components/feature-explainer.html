Feature Explainer

A structured breakdown of the feature, including request path, code samples, and FAQs.

TL;DR

This feature introduces a new authentication middleware that validates JWT tokens on every request, checks refresh token validity, and gracefully handles expired sessions with automatic re-authentication prompts.

Request Path

1
Client sends request with JWT in Authorization header
▶
The client includes the JWT token in the Authorization: Bearer <token> header. If no token is present, the middleware immediately returns a 401 Unauthorized response. The middleware extracts and decodes the token for validation.
2
Middleware validates token signature and claims
▶
The middleware verifies the JWT signature using the configured public key, checks the exp claim for expiration, validates the iss (issuer) and aud (audience) claims, and ensures the sub claim maps to an active user.
3
Token is expired - check refresh token
▶
If the JWT is expired but the refresh token cookie is present and valid, the middleware automatically issues a new JWT and attaches it to the response headers. The client can then retry the original request with the new token.
4
Request proceeds to route handler
▶
After successful validation, the middleware attaches the decoded user payload to the request object as req.user. The route handler can access user ID, roles, and permissions directly from this object.

    httpOnly: true,
    secure: true,
    sameSite: 'strict',
  },
};
// auth.middleware.ts
import { authConfig } from './middleware.config';

export async function authMiddleware(req, res, next) {
  const token = extractBearerToken(req);
  if (!token) {
    return res.status(401).json({ error: 'No token provided' });
  }

  try {
    req.user = await verifyJWT(token, authConfig.jwt);
    next();
  } catch (err) {
    if (err.name === 'TokenExpiredError') {
      return handleExpiredToken(req, res, next);
    }
    return res.status(401).json({ error: 'Invalid token' });
  }
}
// app.ts
import { authMiddleware } from './middleware/auth.middleware';

// Apply to protected routes
app.use('/api', authMiddleware);

app.get('/api/profile', (req, res) => {
  res.json({
    id: req.user.sub,
    roles: req.user.roles,
  });
});

Frequently Asked Questions

What happens if both JWT and refresh token are expired? +
The middleware returns a 401 Unauthorized response with a specific error code SESSION_EXPIRED. The client should redirect the user to the login page. The refresh token cookie is cleared server-side to prevent reuse.
Can this middleware handle role-based access control? +
Yes. The decoded JWT payload includes a roles array. You can create a companion requireRole(...roles) middleware that chains after authMiddleware to enforce role-based access on specific routes.
How are token rotations handled during refresh? +
When a refresh token is used, the old refresh token is invalidated and a new one is issued (refresh token rotation). This prevents replay attacks. If a previously used refresh token is presented, all refresh tokens for that user are invalidated as a security precaution.
What is the performance impact of JWT validation on every request? +
JWT validation is a purely computational operation (signature verification) with no database lookups. Typical validation takes under 1ms. For high-throughput scenarios, consider caching decoded tokens in a short-lived LRU cache keyed by the token JTI claim.
